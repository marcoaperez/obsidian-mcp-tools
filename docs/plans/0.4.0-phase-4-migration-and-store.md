## Goal

Close the gap between `0.4.0-alpha.4` (functionally complete, all 20 tools + native semantic search verified) and `0.4.0` stable shipped through the official Obsidian community store.

The four work streams:

1. **Migration UX** — first-load detection of legacy 0.3.x state, opt-in modal to rewrite `claude_desktop_config.json`, delete the orphan `mcp-server-*` binary, suggest Local REST API uninstall.
2. **Client config UI** — three "Copy config" generators (Claude Desktop, Claude Code, generic streamable-http for Cursor/Cline/Continue) plus the access-control bearer-token surface (D6 in design).
3. **Claude Desktop integration UX** — Node.js detection + opt-in `npx mcp-remote` pre-warm (D7).
4. **Release engineering** — CI workflow simplification (no more cross-platform binary builds), docs pass (README/CHANGELOG/CLAUDE.md), 0.4.0-beta.1 → 0.4.0 stable, store PR #11919 update, upstream README PR + Discord DM (gated on store acceptance).

## Non-goals

- Reranking and HNSW (Phase 5 / 0.5.0).
- Second embedding model (`bge-small-en-v1.5`) — Phase 5.
- A managed signing/notarization pipeline for the (now retired) server binary.
- Backwards-compatible coexistence of 0.3.x and 0.4.0 in the same vault — Obsidian disallows multiple versions of the same plugin id by design.

## Tasks

Numbered T1-T14. Each task is a logical unit suitable for one commit (or a small PR). Execution order respects dependencies.

### Block A — Migration core (backend logic, testable without UI)

**T1. Detect legacy 0.3.x state.**
Pure function `detectLegacyInstall(plugin): { hasLegacyBinary: boolean; legacyBinaryPath?: string; hasLegacyConfigKeys: boolean; legacyConfigEntry?: { name: string; command: string } }`. Inputs: `plugin.loadData()` (looks for `installLocation` or `platformOverride` keys) + filesystem probe at the platform-specific binary install path. Output drives the migration modal. Located in `features/migration/services/detect.ts`. Tests cover macOS/Windows/Linux paths via `os.homedir()` mock.

**T2. Migration plan + executor.**
Pure function `planMigration(state)` returns a list of `MigrationStep[]` (rewrite Claude config, delete binary, suggest LRA uninstall, prune legacy settings keys). Executor runs the steps, each with an `actuallyApply(): Promise<{ ok: boolean; error?: string }>`. Each step is independently opt-in. `claude_desktop_config.json` is backed up to `.backup` before rewrite. Tests cover dry-run + each step in isolation against the in-memory mock fs.

**T3. New `updateClaudeConfig` shape.**
Replace the 0.3.x payload (`{ command, args: [<binary path>], env: { OBSIDIAN_API_KEY } }`) with the 0.4.0 payload (`{ command: "npx", args: ["-y", "mcp-remote", "http://127.0.0.1:<port>/mcp", "--header", "Authorization: Bearer <token>"] }`). Token is read from `mcpTransport.bearerToken` in `data.json`. Port is read from the live `McpTransportState`. Backwards-compatible signature with the existing tool-toggle code path. Existing 8-test invariant suite (`config.test.ts`, added in 0.3.10 for issue #11) is updated to assert the new shape AND that the rewrite preserves unrelated `mcpServers` entries.

### Block B — Client config generators

**T4. Pure config generators.**
`features/client-config/services/generators.ts` exposes:
- `claudeDesktopConfig({ url, token }): JsonObject` — npx mcp-remote shape.
- `claudeCodeConfig({ url, token }): JsonObject` — `{ type: "http", url, headers: { Authorization: "Bearer …" } }`.
- `streamableHttpConfig({ url, token }): JsonObject` — `{ type: "streamable-http", url, headers: { Authorization: "Bearer …" } }` (covers Cursor / Cline / Continue / Windsurf / VS Code with per-client notes).

Each returns a JSON object; UI serializes with `JSON.stringify(…, null, 2)`. ArkType validation on every input. Pure, no side effects, fully unit-testable.

**T5. Settings UI: Access Control section.**
Svelte component in `features/mcp-transport/components/AccessControlSettings.svelte`:
- Bearer token field (password-style by default, "Show" toggle, "Copy", "Regenerate" with confirm modal).
- Three "Copy config" buttons that call the T4 generators and `navigator.clipboard.writeText()` the JSON.
- "Auto-write Claude Desktop config" toggle (opt-in, default OFF) — when ON, port changes or token rotations trigger `updateClaudeConfig` (T3).

**T6. Token rotation flow.**
Wire "Regenerate" button to: confirm modal → `crypto.randomBytes(32).toString("base64url")` → write to `data.json` → if auto-write is ON, immediately call T3 → notify user. Existing token consumers (the HTTP transport middleware) re-read on next request, no restart needed.

### Block C — Migration UX surface

**T7. Migration modal (Svelte).**
`features/migration/components/MigrationModal.svelte`. Shown on plugin load if `detectLegacyInstall()` returns any positive signal AND the user has not previously dismissed it (persist "skipped" flag in `data.json`). Body matches the design: three opt-in checkboxes (rewrite Claude config / delete legacy binary / suggest LRA uninstall) + [Migrate] / [Skip for now] / [Learn more] buttons. "Learn more" opens the upgrade section of the README in the default browser.

**T8. First-load wiring.**
In `main.ts`, after all features finish setup, call `detectLegacyInstall()` once. If positive, queue the modal for after the Obsidian workspace is ready (use `app.workspace.onLayoutReady`). Track a single-shot guard so the modal does not re-open in the same Obsidian session.

### Block D — Claude Desktop integration UX (D7)

**T9. Node.js detection service.**
`features/mcp-transport/services/nodeDetect.ts`: spawn `node --version` via `child_process.exec`, parse with semver, cache result for the session. Read-only, no network. Tests use the existing fake-binary pattern (real shell scripts in tmpdir, see `status.integration.test.ts`).

**T10. mcp-remote pre-warm.**
`features/mcp-transport/services/preWarm.ts`: runs `npx -y mcp-remote@latest --help`, streams progress to UI, on success caches `{ version, lastChecked }` in `data.json`. UI lives next to T5 in the Access Control section. Idempotent — re-running is safe.

### Block E — Release engineering

**T11. CI workflow simplification.**
`.github/workflows/release.yml` retains only the plugin-asset path: `bun install --frozen-lockfile` → `bun test` → `bun run check` → `bun run build` (plugin only) → `bun run zip` → upload `obsidian-plugin-*.zip` + `main.js` + `manifest.json` with SLSA attestation. Removes mcp-server-* cross-platform binary jobs. Adds an on-PR workflow (`.github/workflows/ci.yml`) that runs lint + test + check on PRs against `main` and `feat/http-embedded`.

**T12. Documentation pass.** (Partial: alpha-line notice + status update landed; full rewrite deferred to T14.)
- `README.md` — strip 0.3.x references (binary install, Local REST API requirement). Add the 0.4.0 install flow (BRAT now, store post-acceptance), the three copy-config buttons, the migration modal screenshot, the Node.js prerequisite for Claude Desktop only. **Status:** added a "0.4.0 alpha — new architecture" notice in commit batch with `CLAUDE.md`; full rewrite happens at T14 stable cut.
- `CHANGELOG.md` — collapse the alpha.X entries into a single 0.4.0 stable entry preserving the lessons-learned subsections. **Status:** deferred to T14 — alpha entries stay verbatim through beta soak so the alpha→beta diff is reviewable.
- `CLAUDE.md` — update the "Project status" + "Pending work" sections, remove obsolete "Phase 2/3" pending items, add the Phase 4 closure note. Replace the 0.3.x architecture description with the 0.4.0 one. **Status:** "Project status" + "Pending work" updated to reflect 0.4.0-alpha.4 + Phase 4 progress; the architecture section rewrite is part of T14.

**T13. 0.4.0-beta.1 cut.**
After T1-T12 land and the manual E2E matrix (vault TEST + Claude Code + Claude Desktop + at least one of Cursor/Cline) is green: tag `0.4.0-beta.1`, BRAT users on the alpha line auto-receive it. Triage feedback from BRAT for ~1 week.

**T14. 0.4.0 stable.**
After beta soak: tag `0.4.0`, edit store PR #11919 to point at the new manifest, await store acceptance. Once accepted: open README PR upstream + Discord DM to @jacksteamdev (per his stated conditions on issue #79). Mark `0.3.x` line "maintenance only" in handoff + add the deprecation timeline notice to README.

## Test strategy

| Layer | Tooling | Coverage |
|---|---|---|
| Unit | `bun test` | T1, T2, T3, T4, T9, T10 — pure functions and side-effects mocked through the existing test-setup |
| Integration | `bun test` against the existing in-memory mock | T2 + T3 + T7 wire-up, T8 first-load gating |
| E2E | Manual runbook in vault TEST | T5 + T6 + T7 (UI flows), T11 (CI dry-run on a throwaway tag), T13 BRAT smoke |

The existing `packages/obsidian-plugin/src/test-setup.ts` already mocks `FileSystemAdapter`, `Modal`, `svelte` mount/unmount. Phase 4 extends it (≤ ~50 LOC) for `child_process.exec` capturing if T9/T10 testability requires it; otherwise reuses the fake-binary pattern.

## Risk register

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Store PR #11919 reviewer asks for the manifest to point at a tagged stable, not BRAT pre-release | medium | low | T14 happens AFTER 0.4.0 stable tag is up; store PR edit is the last step |
| `npx mcp-remote` first-run delay confuses users on Claude Desktop | medium | medium | T10 pre-warm UX + clear copy in T5 settings + README documents the "first launch is slow" caveat |
| Migration modal misfires on a vault that never had 0.3.x | low | medium | T1's detect function requires positive signals (file-exists OR settings keys present); negatives → no modal |
| Existing 0.3.x BRAT users break when they receive 0.4.0 | medium | high | Migration modal is always opt-in; legacy `installLocation`/`platformOverride` keys become no-ops in 0.4.0 (silently ignored), they do NOT cause settings-load failures |
| `mcp-server-install` feature removal breaks tool-toggle UI which imports `updateClaudeConfig` | medium | low | T3 keeps `updateClaudeConfig` in place but with the new payload; tool-toggle import stays valid |

## Definition of done

All 8 success criteria from the design doc § "Success criteria":

1. ✅ All 20 tools operate via HTTP (validated end-of-Phase-3).
2. ✅ Native semantic search query → result < 100 ms (15 ms measured at end of alpha.4).
3. Fresh install < 60 s end-to-end. **(Validate during T13 beta soak.)**
4. Migration from 0.3.x with no settings/config loss. **(T1-T8.)**
5. Zero ESLint errors from `ObsidianReviewBot`. **(Validate at T14 store PR update.)**
6. 0.4.0 accepted in the Obsidian community plugin store. **(T14.)**
7. Link in `jacksteamdev/obsidian-mcp-tools` README obtained. **(T14, after store acceptance.)**
8. ≥ 5 pieces of community feedback within 2 weeks post-release. **(Tracked post-T14.)**
