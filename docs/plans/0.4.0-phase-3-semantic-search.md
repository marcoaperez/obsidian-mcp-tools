# 0.4.0 — Phase 3: Native semantic search

> **For agentic workers:** RECOMMENDED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Tasks T3 (chunker), T4 (embedder), T5 (store), T7 (Smart Connections provider) are pure logic with no shared state and can be parallelized once T1-T2 (bootstrap + settings schema) complete.

**Goal:** Bring `search_vault_smart` from "Smart Connections plugin required" to "self-sufficient by default". Add a native semantic search backend via Transformers.js + MiniLM-L6-v2, with Smart Connections retained as an opt-in alternative through a tri-state setting. Phase parity with the design doc § Semantic search.

**Architecture:** A `SemanticSearchProvider { search(query, opts) }` interface with two implementations — `NativeProvider` (Transformers.js + local indexer) and `SmartConnectionsProvider` (wraps existing reactive loader). The current `searchVaultSmart` tool handler stops calling Smart Connections directly and dispatches via the provider chosen by settings. The native provider owns its lifecycle (model download, index build, vault file watching). The indexer runs in **Live** mode by default (re-embed on `vault.modify`/`vault.create`/`vault.delete`, debounced); a **Low-power** opt-in scans every 5 minutes for batched re-embedding.

**Tech Stack:** TypeScript strict + `verbatimModuleSyntax`, ArkType 2.x, `@xenova/transformers` (new dependency, ONNX runtime in JS/WASM), `Xenova/all-MiniLM-L6-v2` (single model in 0.4.0 per design D8), Bun test runner, Obsidian Plugin API (`vault.adapter` for binary file I/O, `vault.on('modify'|'create'|'delete')` for live mode).

**Design reference:** `docs/design/2026-04-24-http-embedded-design.md` § Semantic search. Decisions D7 (tri-state), D8 (single model in 0.4.0), D9 (live + low-power dual-mode indexer).

**Branch:** `feat/http-embedded`. Latest commit at the start of Phase 3: `6de78c1` (handoff refresh after T24).

## Task 3: Chunker (pure function)

**Files:**
- Create: `services/chunker.ts` + `chunker.test.ts`

**Contract:** `chunk(content: string, opts?: ChunkOpts): Chunk[]`. `Chunk` has `{ id: string, text: string, heading: string|null, offset: number, contentHash: string }`. Heading-section chunks (split on H1/H2). Section longer than 512 tokens → sliding window 512/64 overlap. Section under 20 tokens → skip. Frontmatter concatenated to first chunk.

**Token counting:** approximate via `text.split(/\s+/).length`. Exact MiniLM tokenizer is overkill for window decisions and slows the chunker significantly. Document the approximation.

**Hashing:** `contentHash = sha256(chunk.text).slice(0, 16)` via `crypto.subtle.digest`.

**Tests:**
1. `single H1 section under 512 tokens → 1 chunk`
2. `H1 + multiple H2 sections → 1 chunk per section, frontmatter merged with first`
3. `long section (>512 tokens) → multiple chunks with 64-token overlap`
4. `section under 20 tokens → skipped`
5. `no headings → single chunk if under 512 tokens, sliding window otherwise`
6. `contentHash is deterministic and changes on text edits`

Commit: `feat(semantic-search): chunker (heading-section + sliding window fallback)`.

```typescript
export interface Embedder {
  embed(text: string): Promise<Float32Array>;

```

**Implementation:**

4. `embedBatch: returns one vector per input`
5. `unload: clears the pipeline; next call re-loads`
6. `idle timer: 60s after last call, pipeline is unloaded`

## Task 5: Store (binary embeddings + JSON index)

**Files:**
- Create: `services/store.ts` + `store.test.ts`

**Contract:**

```typescript
export interface EmbeddingStore {
  init(): Promise<void>;                                // load index + open file
  size(): number;                                       // number of chunks indexed
  upsert(records: EmbeddingRecord[]): Promise<void>;    // append or replace
  delete(filePath: string): Promise<void>;              // remove all chunks for a file
  scan(): AsyncIterable<EmbeddingRecord>;               // for cosine search
  flush(): Promise<void>;
  close(): Promise<void>;
}

export type EmbeddingRecord = {
  chunkId: string;
  filePath: string;
  offset: number;
  heading: string | null;
  contentHash: string;
  vector: Float32Array;  // 384-dim
};
```

**Storage format:**
- `.obsidian/plugins/mcp-tools-istefox/embeddings.bin` — sequential Float32 blocks, 384 × 4 bytes per chunk.
- `.obsidian/plugins/mcp-tools-istefox/embeddings.index.json` — `[{chunkId, filePath, offset, heading, contentHash, byteOffset, byteLength}]`. `byteOffset`/`byteLength` point into the .bin file.
- Format version field: `{ version: 1, records: [...] }`. Bump when changing layout (e.g., D8 future bge-small-en model).

**I/O:** use `app.vault.adapter.readBinary` / `writeBinary` for the .bin file (works through Obsidian's adapter — handles iOS/Android paths correctly even though we're desktop-only at 0.4.0). Use `JSON.parse`/`stringify` for the index.

**Tests:**
1. `init on empty: creates files, size === 0`
2. `upsert + size: 3 records → size === 3`
3. `upsert with existing chunkId: replaces in place (vector + metadata)`
4. `delete by filePath: removes all chunks for that path, size decreases`
5. `scan: yields all records in insertion order`
6. `flush + reopen: state persists across init cycles`
7. `format version mismatch: re-init from scratch, log warning`

Commit: `feat(semantic-search): persistent embedding store (binary + JSON index)`.

**Contract:**

```typescript
export interface SemanticSearchProvider {
  search(query: string, opts: SearchOpts): Promise<SearchResult[]>;
```

**Implementation:**

- `excerpt`: first 200 chars of the chunk text, with the matched heading as prefix if available.

```typescript
function chooseProvider(settings: SemanticSearchSettings, plugin: McpToolsPlugin): SemanticSearchProvider {
  const sc = isSmartConnectionsAvailable(plugin);
  switch (settings.provider) {
    case "native":            return new NativeProvider(...);
    case "smart-connections": return new SmartConnectionsProvider(plugin); // throws actionable error on first call if SC absent
    case "auto":              return sc ? new SmartConnectionsProvider(plugin) : new NativeProvider(...);
  }
}
```

The factory is **synchronous** in choice but the providers themselves are async (model load, store init). `isReady()` reflects the async warm-up.

## Task 9: Indexer — Live mode

**Files:**
- Create: `services/indexer.ts` + `indexer.test.ts`

**Lifecycle:**
- On `setup` (when active provider is `NativeProvider`): schedule full index build (debounced, idle-triggered via `requestIdleCallback` if available, fallback `setTimeout(..., 0)`).
- Subscribe to `app.vault.on("modify"|"create"|"delete")`.
- Per-file debounce: 2s of inactivity since last event for that path → re-chunk + chunk-delta re-embed.
- Chunk-delta: compute new chunks, compare `contentHash` per chunk to existing records; only re-embed changed chunks. Untouched chunks keep their existing vector.
- Deleted file → `store.delete(filePath)`.

**Tests:** use a mock vault with deterministic embedder.
1. `full build on setup: indexes all markdown files`
2. `modify event → re-embed only changed chunks`
3. `create event → embeds new chunks`
4. `delete event → removes all chunks for that path`
5. `debounce: rapid edits within 2s collapse to one re-embed`
6. `unsubscribed on plugin disable`

Commit: `feat(semantic-search): live indexer with chunk-delta re-embedding`.
